Applicability Linux/Windows Servers Version 1.0 (Audit Ready)

1. Purpose

To define the standard procedure for identifying, troubleshooting, and resolving backup failures in the trading infrastructure, ensuring zero data loss and compliance with audit requirements.

2. Scope

This SOP applies to all L1 and L2 Support Engineers monitoring the Daily Operations Dashboard. It covers SQL Dumps, File System Backups, and Mirror Sync issues.

3. Responsibility Matrix

  • L1 Engineer: Detection, Initial Retry, Ticket Logging.
  • L2 Engineer: Log Analysis, Space Management, Script Debugging.
  • IT Manager: Critical Data Loss Reporting, Audit Explanation.

4. Step-by-Step Procedure

Step 1: Verification

Do not rely solely on the dashboard alert. Login to the server and verify the physical file existence in /backup/daily/.

Step 2: Log Analysis

Check the automation logs located at C:\SaXhamOps\Logs\ (Windows) or /var/log/saxham_backup.log (Linux). Look for "Space Insufficient" or "Locked File" errors.

Step 3: Corrective Action
  • If Space Issue: Clear old temp files (refer to SOP-CLEAN-02).
  • If Service Stuck: Restart the SQL Writer / Cron Service.
  • If Network Issue: Ping the NAS storage IP.
Step 4: Manual Execution

Run the backup script manually: ./manual_backup.sh --force. Wait for the "SUCCESS" echo.

Step 5: Documentation

Update the Daily Ops Dashboard with the resolution note. Failure to log the resolution is a compliance violation.

5. Escalation Matrix

If the backup fails after Step 4, escalate immediately via WhatsApp Group "Ops-Critical":

  • Level 1: Shift Lead (Within 15 mins)
  • Level 2: Nitin Pawar (Within 30 mins)
